### 1. Post-Restore Filesystem Consistency Check

**Priority: High — this is the most dangerous silent failure**

A restore can complete without error and still produce an unbootable disk. Today there is
no validation that the data written to the target is filesystem-coherent.

**What to do:**
- After `RestoreEngine` finishes writing, attach the target VHDX read-only
- Run `chkdsk /scan /perf` on the mounted volume (scan only, no repair, no reboot required)
- Parse the exit code: `0` = clean, `2` = changes found (logged), `3` = errors found
- Surface the result in the RestorePage UI and write it to the operation history entry
- If `chkdsk` is not available (WinPE minimal image), fall back to reading the NTFS boot
  sector and verifying the signature `0x55AA` and `BytesPerSector` field

**Files to modify:**
- `src/Chronos.Core/Imaging/RestoreEngine.cs` — add post-restore verification step
- `src/Chronos.App/ViewModels/RestoreViewModel.cs` — surface result in UI
- `src/Chronos.Core/Models/ImageSidecar.cs` (or operation history model) — persist result

**Tests:**
- [x] Restore a known-good VHDX → chkdsk passes → UI shows "Filesystem OK"
- [ ] Restore a deliberately corrupted VHDX → chkdsk reports errors → UI surfaces warning
- [x] Run in WinPE (self-test mode) → falls back to boot-sector check gracefully

### 2. Error Recovery and Cleanup on Operation Failure

**Priority: High — silent partial state is a data hazard**

It's unclear what happens when a backup or restore is interrupted mid-way (I/O error,
USB disconnect, `OperationCanceledException`, out-of-disk-space). The user may end up
with a partial VHDX and no sidecar, with no indication of what state they're in. The next
time they try to restore from that file, it will silently fail or corrupt.

**What was verified / fixed:**
- `BackupEngine.CopyToVhdxAsync`: added `catch when (!copySucceeded)` that deletes the partial
  VHDX file after any failure or cancellation (VHDX handle already closed by `using` block)
- `BackupEngine.CopyToFileAsync`: same pattern — partial raw image deleted on failure
- `BackupOperationsService`: already logs "Cancelled"/"Failed"/"Success" to history on all paths ✓
- VSS snapshots: already disposed in `finally` inside `CopyToVhdxAsync` ✓
- Sidecar `.chronos.json`: already written only after a fully successful copy ✓
- `RestoreViewModel`: added `IOperationHistoryService` injection; `finally` block now always
  logs a "Restore" entry with status "Success"/"Cancelled"/"Failed" and error message
- `RestoreEngine` handles/locks: all cleaned up in `try/finally` on every path ✓
- No partial-file concern for restore (writes to physical disk, not creating a new file) ✓

**Test paths to exercise:**
- [x] Cancel a backup at ~50% → partial VHDX is removed, history shows "Cancelled"
- [ ] Simulate I/O error mid-backup → error surfaces in UI, partial file deleted, history shows "Failed"
- [ ] Out-of-disk-space during backup → graceful error, no partial file left behind
- [ ] Restore cancel → history entry shows "Cancelled" with timestamp
- [ ] Restore success → history entry shows "Restore" / "Success" with bytes processed
- [ ] Restore failure (bad source) → history entry shows "Restore" / "Failed" with error message
- [x] ~~Update `CLAUDE.md` with cleanup/recovery patterns~~ (done — see below)

### 3. End-to-End Restore Validation (Manual)

**Priority: High — must be done before any release claim of "production ready"**

There is no automated test that proves the full loop: backup a live partition → restore to a
target disk → target disk boots to Windows. This is the single most important thing to validate.

**Test matrix (check off each):**
- [x] Full disk backup (system drive, VSS) → restore to identical hardware → boots
- [ ] Partition backup → restore to same partition → boots
- [x] Backup on x64 → restore on x64
- [x] Backup in full Windows → restore in WinPE (bare-metal recovery scenario)
- [x] Backup with compression → restore → boots
- [ ] Backup with compression disabled → restore → boots
- [x] Verify operation on restored backup reports clean
- [x] Operation history entries are correct for each

**Note:** VSS backup of the running system drive is the hardest path. Test this explicitly —
don't only test non-system drives.

### 4. Cross-Sector-Size Restore (512B ↔ 4K)

**Priority: Medium — current error message is correct, implementation is complex**

Restoring a 512-byte-sector image to a 4K-native drive (or vice versa) is currently blocked
with a clear error. The implementation requires:

1. GPT header parser — read protective MBR, GPT header, CRC32 validation
2. LBA translator — `new_lba = (old_lba × old_sector_size) / new_sector_size`
3. Partition alignment validator — all partitions must align to new sector boundaries
4. NTFS boot sector patcher — update `BytesPerSector` field after LBA translation
5. Backup GPT header update (at end of disk)

This is well-scoped. Implement after items 1–3 above are complete.

**Files:**
- `src/Chronos.Core/Imaging/RestoreEngine.cs` — replace the current "blocked" error with the
  translation pipeline
- `src/Chronos.Native/` — GPT parsing structs if not already present
- Tests: restore 512B image to 4K VHD and vice versa (VHD loopback, no physical disk required)

- [ ] 512B → 4K restore succeeds end-to-end
- [ ] 4K → 512B restore succeeds end-to-end
- [ ] Alignment validation catches misaligned partitions and surfaces a clear error
- [ ] Update `CLAUDE.md` with sector-size notes

### 5. ChronosBackup.script — Polish

**Priority: Low — it works, these are improvements**

The current script (`ChronosBackup.script`) works against the customized WinRX/PhoenixPE
environment. Possible improvements:

- **ARM64 option** — add architecture radio/dropdown so ARM64 builds can be selected (the
  build pipeline already produces ARM64 ZIPs)
- **Version display** — parse and display the installed version in the interface after build
- **Explicit system DLL handling** — the script works because the custom PE already includes
  the required DLLs; for portability to vanilla PhoenixPE, a `RequireFileEx` block (per
  `pe-plugin-plan.md`) would make it self-contained. Evaluate whether this is needed.
- **Cache behavior** — `cb_AlwaysDownload` defaults to `True`; consider a smarter
  "download only if newer" check

- [ ] Decide on ARM64 support scope
- [ ] If `RequireFileEx` block is added, update `CLAUDE.md` with the DLL list
